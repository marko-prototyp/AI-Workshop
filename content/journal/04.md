## Wattlog, nine screens.

The session produced a sitemap, wireframes for nine screens, state maps for every screen, and a content sourcing inventory. The sitemap is an in-conversation diagram alongside the rationale that built it. Wireframes and state maps are markdown files in the repo.

<figure class="journal-figure"><img src="../assets/weeks/04/wattlog-wireframe.png" alt="Wattlog wireframe"><figcaption>Wattlog wireframe</figcaption></figure>

The cuts: an onboarding flow, a flat rides list, notifications, a public profile, global search, a generic stats hub. Each one documented with a specific reason it would weaken the product. The rides list is worth pausing on. A chronological list of every ride is easy to build and nearly useless for a self-coached cyclist. If you want to understand your training, you need Calendar. *"If a flat list ever feels necessary, Calendar isn’t doing its job."*

No terminal error states: every failure routes back to something actionable. A training tool can’t afford dead ends that make you feel like you broke something. Autosave on blur, no Save buttons anywhere: the user trains and logs, the data is always safe. Every destructive action through exactly one modal, same shape every time, so it always means the same thing.

The content sourcing inventory settled on zero AI-generated content in v1. For a tool that needs to be credible to someone training seriously, AI-generated copy in those slots would undermine it.

What’s still stuck: web, native mobile, or both. A web tool you open after a ride is a different product from a mobile app you use during one. The answer changes the screen sizes, the interaction model, and where the tool lives in someone’s training routine. No defensible answer yet.

## ClassArcade, built around one flow.

Four screens and two overlays: Class Picker, Main Arcade, New Classroom flow, Edit Roster modal, with the Give Points panel and Classroom dropdown on the Arcade.

<figure class="journal-figure"><img src="../assets/weeks/04/screenshot-2026-05-28-at-15.51.19.png" alt="Class Arcade wireframe"><figcaption>Class Arcade wireframe</figcaption></figure>

The key decision was the New Classroom flow. Two steps: class name first, then roster. At least one student required before it completes. That single constraint makes the empty-student-list state on the Arcade unreachable. It would have needed its own design, its own copy, its own handling. Instead it can’t happen.

The Give Points loop is what the whole app is built around: click Give Points, pick polarity, pick amount, check students, confirm. Everything else on screen gives context to that loop. The left column is where the teacher operates. The right column is what the class watches. Two audiences, one screen. Keeping them on opposite sides is what stops the layout from fighting itself.

Things cut: the ticker strip, per-row edit and delete controls on the Arcade, a raised first-place podium card, a deleted-students history view. The last one became a 30-second undo instead. It handles the real mistake without building a screen for an edge case that almost never comes up.

## Naučimo Hrvatski, spec locked.

The Wednesday session ended with Marin writing trip content by hand. The 1:1 on Friday finished the job.

The big decision: Kenzie doesn’t type anything. Every language interaction in the app uses pre-built answer chips. She picks, not writes. *"We decided not to make Kenzie write anything herself because that is too complicated."* That one call removed a pile of problems — no runtime AI needed to validate her Croatian, no diacritic keyboard issues, and a much simpler build. The tradeoff is some depth in active recall, but each scene ramps up across scenes to compensate.

Six wireframes followed from that. The spec is clean enough to build from.
