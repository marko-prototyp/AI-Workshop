---
num: "11"
title: "Find what's wrong. Pick three to fix."
phase: "Test → Critique"
capability: "Hostile QA; multi-lens critique; impact-ranked fix lists; designer-grade failure-state design"
aim: "Whose eyes are the harshest on this project? Through what lens does it look worst? Look at it that way on purpose. The break list and the critique list converge into one fix list."
toolkit: "Opus in Claude.ai chat. Critique depth determines what gets fixed; reasoning quality changes the answer."
exercise:
  - "Run hostile QA on your own project (first 20 min)"
  - "Build at least one designed failure state (next 20 min)"
  - "Run the 5-lens critique pass (next 15 min)"
  - "Build the unified prioritized fix list (last 5 min)"
prompts:
  - title: "Find the breaks"
    preamble: "What's the worst user of your app? Slow connection, wrong locale, wrong file format, hostile inputs. Try to predict three breaks before you copy."
    body: |
      Pretend you're a hostile QA tester. Walk through my app [paste relevant code or describe the flow] and list every way you could break it. Bad input, slow network, wrong file format, empty state, no API response, weird locale, unusual screen size, anything you can think of. Rank by likelihood of happening in real use, not by severity. For each, name what would visibly happen right now (the unhandled case).
  - title: "Build the broken state"
    preamble: "Failure UI is real UI. What does your project look like when it's failed gracefully? Not apologetic, honest."
    body: |
      For [break case], design the failure state. The user should know what failed, why (if relevant), and what they can do. Don't make it apologetic. Match my visual direction "[name]" and banned moves [paste]. Then build it and slot it into the existing code at the place where the failure can actually happen. Show me the trigger condition and the user-visible result side by side.
  - title: "Multi-lens critique"
    preamble: "Critique is a prompt you write to be uncomfortable. What's the question you'd rather not ask? That's probably the one you should."
    body: |
      Critique my project across 6 lenses, one at a time:
      1. Accessibility — contrast, focus, semantic HTML, alt text, keyboard navigation
      2. Visual hierarchy — what does the eye go to? What gets lost?
      3. Copy — does it sound human? Reference my banned-words: [paste]
      4. Consistency — spacing, type, alignment, state transitions
      5. Stock-UI drift — does it feel generic? Compare against my Figma comps.
      6. Device fit — does it work on the device my real user actually has?

      For each, give me 3 specific issues with locations (selectors or line numbers), ranked by severity. Only what to fix, not what's already good.
  - title: "Rank by impact"
    preamble: "Critique always finds twenty things. You have time for three. Which three change the project most?"
    body: |
      Of the issues you raised across the breaks list and the critique lenses, which three would have the largest visible impact if I fixed them in the next 30 minutes? Don't pick the easy ones. Pick the ones that change my project most. For each: name the issue, name the fix, estimate the time to fix.
outputs:
  - "Hostile QA list"
  - "At least one designed failure state shipped"
  - "5-lens critique with locations"
  - "Prioritized fix list with top three impact items"
watch:
  - "Any failure state that's been deferred from earlier sessions ships this week. No more punting."
  - "Don't let people fix easy things first. Must-fixes are usually harder."
  - "Device fit catches things the laptop never showed. Run the test on the right phone."
