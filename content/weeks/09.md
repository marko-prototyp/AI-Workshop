title: "Secondary views. Real states."
phase: "Build"
capability: "State management; designer-led interactivity; state-by-state Figma-to-code; localStorage for designer-grade persistence"
aim: "A hover state isn't the interesting one. What's the empty state? What does your app look like the first second a user opens it with nothing loaded?"
toolkit: "Sonnet in Cowork. Files are on disk now; multi-file state work and persistence happen here."
exercise:
  - "List every state every screen can be in"
  - "Add remaining views and route between them"
  - "Wire up one meaningful interaction"
  - "Persist anything that should survive a page reload"
  - "Commit after each working state"
prompts:
  - title: "State inventory"
    preamble: "Most screens get the default state designed and skip the rest. What does your app look like with nothing loaded? With everything broken?"
    body: |
      For my [screen name], list every state it can be in: default, loading, empty, error, success, hovered, focused, anything else specific to this screen. For each state: what should be visible, what should be hidden, and how the user moves between them. Don't build anything yet. Just the list. At the end, tell me which states are most likely to be skipped and why I shouldn't skip them.
  - title: "Adding interactivity (designer-led)"
    preamble: "Before you ask Claude to code an interaction, can you describe it to a non-engineer in plain English? If you can't, Claude won't get it right either."
    body: |
      I want [interaction]. Here are the two Figma states:
      [Attach 'before']
      [Attach 'after']

      Describe step by step what should happen between them in plain English first, not code. Include: what triggers it, what animates, what stays still, what should happen if I trigger it twice fast, what should happen if the user closes the tab mid-interaction. After I confirm, implement with vanilla JS. Match the visual character of my project.
  - title: "When something breaks"
    preamble: "What's your current move when Claude breaks something after working code? Rebuilding from scratch is the wrong answer."
    body: |
      After the last change, [thing] stopped working. Don't rebuild the whole artifact. Identify what broke and patch only that. If you're not sure what changed, ask me to paste the last working version. Show me the smallest possible fix.
  - title: "Persistence — where does the data live?"
    preamble: "When the user closes the tab and comes back, what should still be there? That's the question localStorage answers."
    body: |
      For each screen with user-generated data, decide: does it need to survive a page reload? If yes, walk me through saving to and reading from localStorage. Show me the keys you'll use, what the empty state looks like before anything's saved, and what happens if the saved data is corrupted or partial. Treat me like someone who's never written JavaScript persistence before. After it works locally, tell me what would break if I cleared my browser cache and how to handle it.
outputs:
  - "All screens exist and route between each other"
  - "One meaningful interaction working"
  - "Empty, loading, and error states for the main view at minimum"
  - "Persisted state survives a page reload where it should"
watch:
  - "Artifacts get hairy this week. Commit before each new state."
  - "Stock-UI drift creeps back in interactions."
  - "Loading states get skipped most. Insist on at least one."
  - "Persistence: localStorage is enough for one-user private projects. No need for a backend."
