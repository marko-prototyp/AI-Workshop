---
num: "10"
title: "Connect the API."
phase: "Build → Wire"
capability: "API basics for designers; key management; stub mode; failing gracefully"
aim: "The first API call you make will fail. Before you write it: what should fail loudly versus silently? What does the user see while the call is in flight?"
toolkit: "Sonnet in Cowork or Claude Code. Real env vars, real file work; Claude Code if you're comfortable, Cowork otherwise."
exercise:
  - "Make one real API call from inside your project"
  - "Add a stub-mode toggle that returns hardcoded data"
  - "Move the API key out of committed code"
  - "Design and build the loading state and the failure state"
prompts:
  - title: "First call (designer-style)"
    preamble: "Before you copy: read the docs of the API you need. Find the smallest example call. Try writing the prompt that asks Claude to translate that example into your project's code."
    body: |
      I'm a designer wiring up [API name] for the first time. My app is plain HTML/CSS/JS. Show me the smallest possible call I can make to prove it's working. Hardcode an API key for now — locally only, we'll handle secrets after it works. Once the call returns something, stop and let me look at the raw response before we touch the UI. Tell me what each part of the response means and which parts I actually need.
  - title: "Stub mode"
    preamble: "Real API calls cost money and burn time. Stub mode lets you build UI without burning calls. Where would the switch live in your app?"
    body: |
      Add a 'use stub' switch to my app. When stub mode is on, instead of calling the real API, return a hardcoded response I provide. The hardcoded response should match the shape of the real API so my UI doesn't know the difference. Show me where the switch lives in code, how to flip it from the browser console, and how to add new stub responses for testing different cases later.
  - title: "Don't commit the key"
    preamble: "If you pushed your code to GitHub right now, would your API key go with it? If yes, the fix is one prompt away."
    body: |
      Walk me through moving my API key out of my code and into a place GitHub won't see. I'm planning to deploy to [Vercel / Netlify]. Cover both: where the key goes locally (so the app still works on my machine) and where it goes in the deploy host (so the deployed app can use it). After I follow your steps, tell me how to confirm the key is no longer in my repo. If it ever was committed, tell me how to scrub it from history.
  - title: "The graceful failure"
    preamble: "When the API doesn't respond, what should your user see? 'Error 500' is not an answer."
    body: |
      Design the failure UI for when my API call fails or times out. The user should know what failed, why, and what they can do about it. Don't make it apologetic. Match my visual direction "[name]" and banned moves [paste]. Cover: timeout, server error, rate limit, no internet, malformed response. Show me the HTML and where each case slots into the existing app.
outputs:
  - "One real API call working"
  - "Stub mode toggleable"
  - "API key out of the repo"
  - "Loading state and failure state for the call"
watch:
  - "Most of the cohort will have API anxiety. Take the time. Some calls land more easily than others."
  - "Anyone whose project doesn't need an API doesn't sit out. Pair-program someone else's call, or use the hour for data-loading work that's API-shaped — data in, view out."
  - "If anyone commits a key, catch it before they push."
