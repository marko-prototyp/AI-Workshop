title: "Design the main view. Figma or Claude Design."
phase: "Design → Build (Bridge)"
capability: "Two design paths: Figma-to-code or Claude Design; design system consolidation; tokens; handoff to Claude Code"
aim: "Your concept artifact validated the structure. Now make it visual. Before you open any tool, you're running the Design System Document prompt — that's the session's real first move, both paths. From there: control every pixel, or write the brief that replaces them."
toolkit: "Figma path: Sonnet in Claude.ai chat with Artifacts. Claude Design path: claude.ai/design. Both produce a designed main view and a tokens.css. Both hand off to Claude Code."
exercise:
  - "Start here, both paths: run the Design System Document prompt. Everything from W01–W05, consolidated into one file. This is what both paths build from."
  - "Choose your path. Figma: comp the main view at the device your real user holds. Claude Design: open claude.ai/design, upload your Design System Document during onboarding, generate the main screen."
  - "Figma path: run the Figma → Claude prompt with your W05 artifact as structural reference. Iterate to match the comp's character. Extract tokens.css from the build."
  - "Claude Design path: during onboarding, upload your Design System Document and W05 smoke test artifact. Describe the main screen. Refine through conversation and inline comments until it matches your intent. Use the built-in handoff to pass to Claude Code."
  - "Both paths: lock your tokens.css before W07."
prompts:
  - title: "Generate the Design System Document"
    preamble: "Both paths start here. If your visual direction is vague, Claude Design returns something generic — and your Figma drifts too. What would make your banned-moves list sharper than it is right now?"
    body: |
      I'm finishing the planning phase of my project and about to start designing. Consolidate everything below into a Design System Document — a single file I'll reference for the rest of the build.

      Project brief: [paste]
      Visual direction: [paste]
      Banned moves: [paste]
      Token file: [paste tokens.css, or "propose a token system from the visual direction above"]
      Wireframes: [paste per-screen descriptions from W04]
      Real content per screen: [paste from W05]
      State map per screen: [paste from W04]

      The document should cover: the visual system in full (colors, type scale, spacing, radii), the art direction in plain language, the banned moves as hard constraints, and a screen-by-screen breakdown with layout hierarchy and real content. Be specific enough that Claude can generate each screen consistently without defaulting to generic SaaS aesthetics. Where anything above is too vague to execute from, ask me one question at a time before writing that section.
  - title: "Figma → Claude (use this every time)"
    preamble: "What's the new information in the comp that the artifact didn't have? Style, exact type, real spacing. Your banned-moves list goes in here too."
    body: |
      [Attach Figma screenshot]

      I have a W05 concept artifact [link or paste] that validated the structure of this screen. Now build this Figma as a single HTML file at high fidelity. The artifact tells you what's on the screen and how it connects. The Figma tells you what it looks like.

      Important rules:

      1. Use these design tokens as CSS variables: [paste, or "extract from this build and define them"]
      2. Use these exact fonts: [Google Fonts links]
      3. Match spacing and proportions of the comp. Don't normalize.
      4. The visual direction is "[name]" — [3-word feel]
      5. Banned moves: [paste relevant items]
      6. Target device: [mobile / tablet / desktop — name what your real user holds]

      If something seems unusual, don't smooth it out. Ask me. Unusual is often the point. Don't add sections I didn't design.
  - title: "Refining a built section"
    preamble: "When Claude's close but not quite, what's the difference between asking for 'better' (vague) and naming the specific gaps (sharp)?"
    body: |
      The [section name] is close but not matching my Figma. Specifically:
      - [Difference 1]
      - [Difference 2]
      - [Difference 3]

      Make only these changes. Don't restructure. Don't normalise. If you think one of my asks is wrong, say so before changing it.
  - title: "Extract tokens from the build"
    preamble: "Claude built your screen but didn't formalise the system. Pull the tokens out so you can change one variable and see it propagate everywhere."
    body: |
      Look at the main view you just built from my Figma. Pull out the implicit design tokens — colors, type scale, spacing scale, radii — and give them back to me as a tokens.css using CSS custom properties. Use semantic names where it makes sense (e.g., --color-ink, --color-rule, --space-3). Comment the file so a designer reading it understands the system. If anything in the build feels off-system or inconsistent, flag it so I can decide whether to keep it or normalise it.
  - title: "Claude Design — describe the main screen"
    preamble: "What's different about describing a screen in words versus sketching it? You're writing a brief Claude Design executes from. Where would it fail if your description was vague?"
    body: |
      Build the main view of my project.

      The design system is set up from the files I uploaded during onboarding. Here's the screen: [describe what the screen shows, what actions are possible, what the visual hierarchy should be — reference your wireframe from W04 and real content from W05]. The visual direction is "[name]" — [3 descriptors]. Banned moves: [paste the most important ones].

      Build the full screen layout first. I'll refine section by section after. If anything in my description seems unusual, ask me before smoothing it out. Don't add sections I didn't describe.
  - title: "Claude Design — correct the drift"
    preamble: "Claude Design will default to what it's seen before. What would your banned-moves list catch first — the corners, the colors, the type weight?"
    body: |
      The [element or section] is drifting from my direction. Specifically: [what's wrong — name it precisely]. Don't rebuild the screen. Fix only that. If fixing it would change anything else, tell me before you do it.
