num: "06"
title: "Design the main view. Make your first skill."
phase: "Design → Build (Bridge)"
principle: "A skill beats a brief. A brief describes your direction and leaves the model room to average toward stock. A skill hands it your tokens, type, and banned moves as a contract, every time."
capability: "Consolidate your direction into one design system; write your first reusable skill; generate the main view and diagnose it"
aim: "Fold everything you've decided into one design system, turn your banned moves into your first reusable skill, then generate the main view from it."
toolkit: "Claude chat · Claude Code · Claude Design (optional renderer)"
tools:
  - Claude chat
  - Claude Code
flow:
  - Consolidate
  - Make the skill
  - Generate & diagnose
before: "Have your W01–W05 work in one place: the brief, the visual direction, the banned moves, the wireframes, and the real content. You'll fold them into one file in the first step."
exercise:
  - "Consolidate W01–W05 into one design system in Claude chat."
  - "Write your first reusable skill, anti-stock, from your own banned moves."
  - "Write a tight brief, generate the main view, and run anti-stock on what comes back."
prompts:
  - title: "Consolidate your direction into one design system"
    tool: "Claude chat"
    does: "Fold everything from W01–W05 into one file. Your brief reads from this, and so does every screen after it."
    preamble: "Where is your visual direction still vague? That's exactly where the model will reach for its default."
    body: |
      I'm finishing the planning phase of my project and about to start designing. Consolidate everything below into one Design System Document I'll reference for the rest of the build.

      Project brief: [paste]
      Visual direction: [paste]
      Banned moves: [paste]
      Colors, type, and spacing so far: [paste any values you've set, or "propose them from the visual direction above"]
      Wireframes: [paste per-screen descriptions from W04]
      Real content per screen: [paste from W05]

      Cover: the full visual system (colors, type scale, spacing, radii), the art direction in plain language, the banned moves as hard constraints, and a screen-by-screen breakdown with layout hierarchy and real content. Be specific enough that the model can generate each screen consistently without defaulting to generic SaaS aesthetics. Where anything above is too vague to build from, ask me one question at a time before writing that section.
    gives: "One design system: the visual system, the art direction, the banned moves, and a screen-by-screen breakdown."
  - title: "Make your first skill: anti-stock"
    tool: "Claude chat"
    does: "Turn the banned moves you've pasted into five prompts into one reusable skill. It reviews a design for the AI stock look. It doesn't generate."
    preamble: "What's the exact stock move you keep catching and correcting by hand? Name it. That's what the skill is for."
    body: |
      I want to build a reusable skill, a SKILL.md the model loads automatically when it's relevant, so I stop re-pasting the same constraints into every prompt.

      What this skill is for: anti-stock. Catch the AI stock look in a generated design, or pressure-test a brief before it generates one. A review pass, not a generator.
      The material it holds: the banned moves and visual direction from the design system we just built in this conversation. (Starting a fresh chat? Paste them here.)

      Write it as a SKILL.md with:
      - YAML front-matter: name (kebab-case) and description. The description is one or two sentences naming exactly when the model should use this skill. Be specific about the situations, because that's what triggers it.
      - A short body: what the skill does, the rules in plain language, the output format, and one short example.

      Two hard rules. Use only the constraints I pasted. Don't invent rules, don't add defaults you think are sensible, don't soften my banned moves. If something I pasted is too thin to write a section from, ask me one question at a time before writing that section.
    note: "This is your first skill, so keep its scope to your own project. Your next two are probably the ones you'll reach for most: the rules behind a screen you admire, and your generation defaults. Don't build them on a schedule. Build each one the moment you catch yourself re-pasting it."
    gives: "anti-stock: an installed skill that catches the AI look, so you stop re-pasting your constraints."
  - title: "Generate the main view, then run anti-stock on it"
    tool: "Claude chat, then your builder"
    does: "Write the tightest brief you can, generate the main view, then turn anti-stock on the result. The skill carries your direction. The builder only renders it."
    preamble: "Where would a vague line let the builder reach for its default? Close it before you generate, not after."
    body: |
      I'm about to generate the main view of my project. A builder renders a brief, it won't reason about it or run my skills, so I want the tightest possible brief first.

      We built my Design System Document and my anti-stock skill together earlier in this conversation, so you already have the wireframes, the real content, the visual direction, the banned moves, and the skill. (Starting a fresh chat? Paste the document and load the skill first.) The screen I'm designing first is the main view: [name it].

      Do three things, all from the document:
      1. Draft the screen brief: what this screen shows, what actions are possible, the visual hierarchy. Use the real content from the document, not placeholders.
      2. Pick the few banned moves most likely to break on this specific screen and state them as hard constraints.
      3. Run my anti-stock skill on your own draft. Where does it leave room for a stock default? Tighten those gaps.

      Ask me one question at a time wherever the document is too thin to design from. When it's tight, give me the finished brief as a single block I can paste straight into my builder, with no commentary around it.
    note: "Then generate, and review what comes back: \"Run my anti-stock skill on this comp. [paste the screenshot, and the exported code if you have it]. Name every AI stock tell, and for each one name the specific fix in my direction's terms. Don't rebuild anything. This is a diagnosis I'll carry into the build.\" Claude Design is one renderer you can use here. It won't run your skill, so harden the brief first and run anti-stock on whatever it returns. You build this in W07, from your design system and skill."
    gives: "A main-view comp, and an anti-stock diagnosis of every place it drifted from your direction."
outputs:
  - "One design system: everything from W01–W05 in a single file."
  - "Your first skill, anti-stock, installed and reusable."
  - "A first main-view comp, with a diagnosis of where it drifted."
watch:
  - "Run all three steps in one conversation, then save your design system and skill to your project. Claude keeps them in context now, and every session after this reads them from the project automatically. You never paste the same thing twice. Re-pasting is the smell that you're working around the model instead of with it."
  - "The brief alone won't save you. Where your look resembles a nice modern app, the model's default leaks through. The skill is what holds the line."
  - "A skill is how you stop re-explaining yourself. The signal to make one is plain: you've pasted the same rule into five prompts."
  - "The builder renders, it doesn't reason. It won't run your skill or critique its own output. Do that work in chat."
  - "80% on the first generation is realistic. You fix the last 20% in W07."
