num: "06"
title: "Design the main view. Make your first skill."
phase: "Design → Build (Bridge)"
capability: "Finalize the design instructions.; write and harden the comp brief in Claude chat, generate the comp in Claude Design, reconcile and build in Claude chat and Claude Code"
aim: "Your concept validated the structure. Now you make the look and build the thing, and each step has a place. The brief, your first skill, the review, and the reconcile happen in Claude chat. The comp happens in Claude Design. The build happens in Claude Code. Claude Design makes the look. It doesn't reason about it, so you do the reasoning before and after."
toolkit: "Claude Design · Claude chat · Claude Code"
exercise:
  - "Run the Design System Document prompt first, in Claude chat. Everything from W01–W05 in one file."
  - "Author one skill, `anti-stock`, in Claude chat, from your own banned moves and visual direction."
  - "In Claude chat, write the comp brief with the brief-builder prompt. Let Claude run anti-stock on its own draft and tighten the gaps. When the brief is finished, paste it into claude.ai/design and generate. Nudge it visually until the comp holds."
  - "Bring the comp into Claude chat as a screenshot, and as exported code if you can get it. Run the anti-stock skill on it again, this time on what Design actually produced. That's your diagnosis."
  - "In Claude chat, reconcile the comp against your brief by hand. Pull the real tokens, diff against the document, write the reconciled v2 and a locked tokens.css."
  - "In Claude Code, build the main view from the comp, the locked tokens, and the anti-stock fixes. Correct it against the comp. Lock the tokens before W07."
prompts:
  - title: "Generate the Design System Document · Claude chat"
    preamble: "The brief reads from this one file, and so does your reconcile later. If your visual direction is vague, every screen after it is generic. What would make your banned-moves list sharper than it is right now?"
    body: |
      I'm finishing the planning phase of my project and about to start designing. Consolidate everything below into a Design System Document — a single file I'll reference for the rest of the build.

      Project brief: [paste]
      Visual direction: [paste]
      Banned moves: [paste]
      Token file: [paste tokens.css, or "propose a token system from the visual direction above"]
      Wireframes: [paste per-screen descriptions from W04]
      Real content per screen: [paste from W05]
      State map per screen: [paste from W04]

      The document should cover: the visual system in full (colors, type scale, spacing, radii), the art direction in plain language, the banned moves as hard constraints, and a screen-by-screen breakdown with layout hierarchy and real content. Be specific enough that Claude can generate each screen consistently without defaulting to generic SaaS aesthetics. Where anything above is too vague to execute from, ask me one question at a time before writing that section.
  - title: "Write your first reusable skill · Claude chat"
    preamble: "You've watched these get used all workshop. Now you make one. You've pasted your banned moves into every prompt for five sessions. That's the signal. The description field is what makes a skill trigger, so name the exact moment you'd want it."
    body: |
      I want to build a reusable skill — a SKILL.md file Claude loads automatically when it's relevant, so I stop re-pasting the same constraints into every prompt.

      What this skill is for: [name + one line, from the row below]
      The material it should hold: [paste your own work, from the row below]

      Write it as a SKILL.md with:
      - YAML front-matter: name (kebab-case) and description. The description is one or two sentences naming exactly when Claude should use this skill — be specific about the situations, because that's what triggers it.
      - A short body: what the skill does, the rules in plain language, the output format it should produce, and one short example.

      Two hard rules. Use only the constraints in the material I pasted. Don't invent rules, don't add defaults you think are sensible, don't soften my banned moves. If something I pasted is too thin to write a section from, ask me one question at a time before writing that section.
    note: "What to build this session — anti-stock: catch the AI stock look in any generated design, or pressure-test a brief before it generates one. A review pass, not a generator. Paste your banned-moves list and visual direction. The same prompt makes any skill. Your next two are probably design-analysis (the rules behind a screen you admire) and design-creation (your generation defaults). Don't build them on a schedule. Build each one when you catch yourself re-pasting it and the shape is obvious."
  - title: "Write the comp brief · Claude chat, then paste into Claude Design"
    preamble: "Claude Design renders a brief. It won't help you write one or run your skill. So write the brief here, harden it with anti-stock, then paste the finished block across. Where would a vague line let Design reach for its defaults?"
    body: |
      I'm about to generate the main view of my project in Claude Design. Design builds a comp from a brief, but it won't reason about that brief or run my skills. So I want to write the tightest possible brief here first, then paste it across.

      My material:
      Design System Document: [paste — it's also uploaded to Design at onboarding]
      Wireframe for this screen (W04): [paste]
      Real content for this screen (W05): [paste]
      Visual direction: "[name]" — [3 descriptors]
      Full banned moves: [paste]

      Do three things:
      1. Draft the screen brief: what the screen shows, what actions are possible, the visual hierarchy. Use my real content, not placeholders.
      2. From my banned moves, pick the few most likely to get violated on this specific screen and state them as hard constraints.
      3. Run my anti-stock skill on your own draft. Where does the brief leave room for a stock default? Tighten those gaps. Leave as little unspecified space as possible.

      Ask me one question at a time wherever my material is too thin to write from. When it's tight, give me the finished brief as a single block I can paste straight into Claude Design, with no commentary around it, in this shape:

      Build the main view of my project.
      The design system is set up from the Design System Document I uploaded during onboarding. [screen description, actions, hierarchy, in real content]. Visual direction: "[name]" — [3 descriptors]. Banned moves: [the few you selected].
      Build the full layout first. I'll nudge it section by section after. If anything seems unusual, ask before smoothing it out. Don't add sections I didn't describe.
  - title: "Run the anti-stock review on the comp · Claude chat"
    preamble: "You hardened the brief with this skill in prompt 3. Now run it on what Design actually produced. The defaults it adds regardless of the brief are what you're hunting. It diagnoses, it doesn't fix."
    body: |
      Review this Claude Design comp against my anti-stock skill. [paste the screenshot, and the exported code if you have it]

      Name every AI stock tell you find — the default corners, the predictable gradient, the centred hero, the SaaS card grid, whatever's there — and for each one, name the specific fix in my direction's terms. Don't rebuild anything. This is a diagnosis I'll carry into the build. I'll decide what to fix.
  - title: "Reconcile the comp against your brief · Claude chat"
    preamble: "The document was a guess made before any pixels existed. The comp is the real thing. Do this reconcile by hand now. Once you've done it a couple of times, you'll know its shape well enough to make it a skill."
    body: |
      Here's the main view comp Claude Design built and I've approved: [screenshot, plus the exported code if you can get it].

      Here's my Design System Document — the plan I wrote before any of this existed: [paste].

      The document was a prediction. The comp is the reality. Reconcile them:

      1. Pull the real system out of the comp — colors, type scale, spacing, radii — as semantic CSS custom properties. Use my token names where they already exist.
      2. Diff the real system against the document. List every difference in a table: what the document said, what the comp does, which one is right.
      3. For each difference, flag it as an improvement to keep or a drift to fix. Don't decide for me on anything that touches the brand. Ask.
      4. Output two things: a reconciled Design System Document (v2), and a clean tokens.css I can lock.

      Don't normalise away anything distinctive without flagging it first.
  - title: "Build the main view · Claude Code"
    preamble: "The locked tokens.css is the one artifact that must travel unchanged. Hand Code the comp as the visual target, the locked tokens, and the anti-stock fixes from your review. Don't let it re-derive anything."
    body: |
      Build the main view of my project.

      Locked tokens.css: [paste]. Treat these values as fixed. Do not re-derive tokens, don't round them, don't rename them.

      Reconciled Design System Document: [paste].

      Visual target — the Claude Design comp: [screenshot, and the exported code if you have it].

      Anti-stock fixes to apply while building: [paste the diagnosis from prompt 4].

      Build the main view to match the comp, using only these tokens and the rules in the document, with the anti-stock fixes applied. Match spacing and proportion to the comp. Where the document doesn't cover something, ask me before inventing it. Don't add sections I didn't design.
  - title: "Correct the build against the comp · Claude Code"
    preamble: "When the build is close but off, name the gap precisely. Vague asks get vague changes. This is the last 20%."
    body: |
      The [element or section] in your build is off from the comp. Specifically: [name it precisely]. Fix only that. Don't rebuild the screen. Don't re-derive the tokens. If fixing it would change anything else, tell me before you do it.
outputs:
  - "One SKILL.md you keep and reuse: `anti-stock`."
  - "A hardened comp brief, written in chat, ready to paste into Design."
  - "A main view comp in Claude Design that holds to your brief."
  - "A reconciled Design System Document (v2) and a locked tokens.css."
  - "The main view built in Claude Code from the comp and the locked system."
watch:
  - "A skill is how you stop re-explaining yourself. The signal to make one is plain: you've pasted the same rule into five prompts in a row. Your banned moves are that rule."
  - "Write the brief in chat, not in Design. Design renders a brief; it won't help you write one. Harden it with your anti-stock skill first, then paste it across. The model's defaults are stronger than any prompt, so the tighter the brief, the less stock leaks through."
  - "Claude Design is a renderer, not a workbench. It won't run your skill, critique its own output, or extract tokens. Bring the comp out as a screenshot, and as exported code if you can get it, then do that work in chat and Code."
  - "Don't make a skill for something you haven't done yet. You reconcile by hand this session. It becomes a skill later, once you know its shape."
  - "The comp is a prediction made visual. The build is the reality. When they disagree, decide each difference yourself. Keep the wins, kill the drift."
  - "Don't let Claude Code re-derive your tokens. Extract them once in chat, lock the file, and that file travels unchanged."
  - "80% on the first generation is realistic. You fix the last 20%."
  - "A Figma-to-code path exists and works the same way: comp first, then build. Nobody here took it. Claude Design is the comp tool now."
